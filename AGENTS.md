# AGENTS.md

# Repository Philosophy

This repository is an experimentation platform for AI audiobook generation.

The goal is not merely to generate audiobooks.

The goal is to continuously discover which combination of:

* document parsers
* OCR engines
* text normalization techniques
* chunking strategies
* prompting strategies
* local LLMs
* TTS configurations
* audio assembly approaches

produces the highest quality audiobook output within strict hardware constraints.

Every major decision must be:

* measurable
* reproducible
* traceable
* comparable

The audiobook is an artifact generated by an experiment.

Experiments are first-class citizens.

---

# Core Product Hypothesis

Kokoro is considered a sufficiently capable baseline TTS engine.

The primary research area of this project is not TTS replacement.

The primary research areas are:

* document understanding
* layout extraction
* chapter detection
* paragraph preservation
* dialogue detection
* chunk orchestration
* pacing optimization
* prosody preparation
* audio quality evaluation

Before introducing new models, attempt to improve output quality through better text preparation.

# Layered Architecture

`app/shared/` holds all domain logic, interfaces, and schemas.

`app/api/` and `app/worker/` orchestrate — they call `app/shared/`, never implement business rules.

`app/llmops/` is read-only — it reads from MLflow and `runs/`; it never writes to them.

**Allowed imports:**
- `app/api/` → `app/shared/`
- `app/worker/` → `app/shared/`
- `app/llmops/` → `app/shared/` (read-only access to schemas and config)

**Not allowed:**
- `app/shared/` → `app/api/`, `app/worker/`, or `app/llmops/`
- `app/api/` → `app/worker/` or vice versa

Business logic must never be duplicated. It belongs in `app/shared/`.

# Hardware Constraints

Primary Development Machine

* CPU only
* 16GB RAM

Secondary Machine

* GTX 1650
* 4GB VRAM
* 8GB RAM

The system must be fully functional on CPU-only hardware.

GPU acceleration is optional.

GPU acceleration is an optimization layer.

The system must never assume multiple models can coexist in GPU memory.

---

# CPU First Strategy

CPU execution is the default execution mode.

Prefer:

* CPU compatible models
* ONNX models
* quantized models
* streaming pipelines
* sequential execution

Avoid introducing solutions that require:

* persistent GPU services
* multiple concurrent models
* large VRAM allocations

A feature that only works on GPU is considered incomplete.

---

# Model Loading Policy

Compute resources are scarce.

Models are temporary workers.

Preferred lifecycle:

Load Model
↓
Execute Task
↓
Collect Metrics
↓
Emit Trace
↓
Unload Model

Assume only one major model can occupy memory at a time.

Long-lived model processes require benchmark evidence.

---

# Local AI Strategy

The platform is fully open-source.

Do not depend on:

* OpenAI
* Anthropic
* Gemini
* paid hosted inference APIs

All AI capabilities should be achievable using local models that can run within project hardware constraints.

Cloud APIs may be used only for benchmarking or comparison experiments and must never become core dependencies.

# Chunking Is A Core Domain

Chunking is not a utility function.

Chunking is a primary research area.

The platform should support experimentation with:

* chapter chunking
* paragraph chunking
* sentence chunking
* semantic chunking
* adaptive chunking
* hybrid chunking

Chunk sizes must not be hardcoded into business logic.

Chunking strategies should be configurable and measurable.

# Experiment Framework

Every pipeline execution is an experiment.

Example:

Experiment A

* Dockling
* No LLM
* Paragraph Chunking
* Kokoro FP32

Experiment B

* Dockling
* Qwen
* Adaptive Chunking
* Kokoro Quantized

Experiments must be declarative.

Prefer configuration-driven experiments over hardcoded implementations.

Example:

```yaml
experiment:
  name: paragraph_chunking_v1

pipeline:
  parser: dockling
  ocr: rapidocr
  chunker: paragraph
  llm: none
  tts: kokoro_fp32
```

# Reproducibility Requirements

Every experiment must be reproducible.

A completed experiment must record:

* experiment_id
* run_id
* git_commit_sha
* git_branch
* execution_environment
* timestamp
* dataset_version

Given a run identifier, the system should be able to reconstruct the exact execution.

Reproducibility is mandatory.

---

# Dataset Versioning

Datasets are versioned assets.

Changes to:

* benchmark books
* evaluation datasets
* test manuscripts
* scoring datasets

must generate a new dataset version.

Experiment results are only comparable when executed against the same dataset version.

---

Prompts are source code.

Prompts belong inside:

* version controlled
* reviewable
* reproducible
* traceable

* template_version
* model_name

Prompt changes must be evaluated through experiments.

# Agent Decision Rules

When making architectural decisions:

1. Prefer observability over simplicity.
2. Prefer reproducibility over convenience.
3. Prefer measurable experiments over assumptions.
4. Prefer CPU-first solutions.
5. Treat prompts as source code.
6. Treat chunking as a core domain.
7. Log every model interaction.
8. Log every prompt execution.
9. Preserve complete experiment lineage.
10. Preserve Git SHA lineage.
11. Preserve dataset lineage.
12. Keep `app/api/` and `app/worker/` thin — orchestration only.
13. Keep business logic inside `app/shared/`.
14. Design every AI component to be replaceable.
15. Design every experiment to be reproducible.
